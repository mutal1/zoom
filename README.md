1.npm init -y
- 기본 설정으로 새 node.js 프로젝트를 초기화하는데 사용
init은 node.js 프로젝트를 생성할 때 사용하는 명령어 해당 종속성 및 메타데이터에 대한 정보가 포함된 구성 파일인 package.json파일을 설정
y 는 yes의 약자로 초기화 프로세스 중에 일반적으로 나타나는 모든 프롬프트에 대해 기본값을 자동으로 수락하도록 npm에 지시내림

package.json
"name": "zoom_clone": 이 줄은 Node.js 프로젝트의 이름을 지정합니다. 이 경우 프로젝트의 이름은 "zoom_clone"입니다.

"version": "1.0.0": 이 줄은 프로젝트의 버전 번호를 나타냅니다. 이 예에서는 버전이 "1.0.0"으로 설정되어 있습니다. 변경 사항을 적용하고 업데이트를 릴리스할 때 시맨틱 버전 관리 규칙에 따라 그에 따라 버전을 업데이트할 수 있습니다.

"description": "": 여기에 프로젝트에 대한 간단한 설명을 입력할 수 있습니다. 현재 설명은 비어 있지만 프로젝트가 하는 일에 대한 간략한 요약을 추가할 수 있습니다.

"main": "index.js": 이 줄은 애플리케이션의 주 진입점을 가리킵니다. 이 경우 주 진입점은 "index.js"로 설정되어 있습니다. 즉, 다른 사람이 패키지를 임포트할 때 Node.js가 이 파일을 찾아 실행을 시작합니다.

"scripts": { "test": "echo \"오류: 지정된 테스트 없음\" && exit 1" }: 이 섹션에서는 npm 명령으로 실행할 수 있는 사용자 정의 스크립트를 정의할 수 있습니다. 이 예제에는 "test"라는 이름의 스크립트가 있습니다. npm test를 실행하면 "오류: 지정된 테스트 없음"을 출력하고 오류 코드 1과 함께 종료됩니다. 이 스크립트를 사용자 지정하여 프로젝트의 테스트를 실행할 수 있습니다.

"keywords": []: 키워드는 사용자가 npm에서 프로젝트를 찾는 데 도움이 되는 설명 태그입니다. 현재 이 프로젝트에 대해 정의된 키워드가 없습니다.

"author": "": 이 필드는 프로젝트의 작성자를 나타냅니다. 현재 비어 있습니다.

"license": "ISC": 이 줄은 프로젝트가 배포되는 라이선스를 지정합니다. 이 경우 프로젝트는 허용형 오픈 소스 라이선스인 ISC 라이선스에 따라 라이선스가 부여됩니다.

package.json 파일은 프로젝트의 메타데이터, 종속성 및 기타 구성을 정의하기 때문에 모든 Node.js 프로젝트에서 중요한 부분입니다. 다른 개발자가 프로젝트를 효과적으로 이해하고 사용할 수 있도록 도와줍니다.

MIT 및 ISC(Internet Systems Consortium) 라이선스는 둘 다 허용적인 오픈 소스 라이선스이며 권한 및 사용 측면에서 많은 유사점이 있습니다. 그러나 둘 사이에는 몇 가지 사소한 차이점이 있습니다.

MIT 라이선스

출처: MIT 라이선스는 가장 오래된 오픈 소스 라이선스 중 하나로 수십 년 동안 널리 사용되었습니다.
조건: MIT 라이선스는 원본 저작권 표시와 라이선스 텍스트가 소프트웨어의 모든 사본 또는 상당 부분에 포함되어야 한다고 요구합니다.
사용법: MIT 라이선스는 다양한 오픈 소스 프로젝트에서 일반적으로 사용되며 소프트웨어 개발 커뮤니티에서 잘 알려져 있고 인정받고 있습니다.

ISC 라이선스:

출처: ISC 라이선스는 MIT 라이선스에서 파생된 관대한 오픈 소스 라이선스입니다. ISC(Internet Systems Consortium)에서 만들었으며 OpenBSD 운영 체제에 포함되어 있기 때문에 "OpenBSD 라이선스"라고도 합니다.
조건: ISC 라이선스는 MIT 라이선스와 거의 동일한 조건을 갖습니다. MIT 라이선스와 마찬가지로 원본 저작권 표시와 라이선스 텍스트가 소프트웨어의 모든 사본 또는 상당 부분에 포함되어야 합니다.
사용법: ISC 라이선스는 MIT 라이선스에 비해 덜 일반적이지만 여전히 일부 오픈 소스 프로젝트, 특히 ISC 또는 OpenBSD 커뮤니티와 관련된 프로젝트에서 사용됩니다.

요약:
MIT와 ISC 라이선스의 실질적인 주요 차이점은 역사적 기원과 관련된 조직입니다. 법적인 관점에서 이들은 거의 동일하며 소프트웨어 사용자에게 유사한 자유를 제공합니다. 개발자는 개인 선호도 또는 자신이 기여하는 프로젝트의 라이선스 내역에 따라 둘 중 하나를 선택하는 경우가 많습니다. 궁극적으로 두 라이센스 모두 허용되며 개발자가 코드를 자유롭게 사용, 수정, 배포 및 프로젝트에 통합할 수 있습니다.

2.npm i nodemon -D

npm i nodemon -D 명령은 Node.js 프로젝트에서 nodemon 패키지를 개발 종속성으로 설치하는 데 사용됩니다.

npm: Node.js 프로젝트에서 패키지(라이브러리)를 설치하고 관리할 수 있는 도구인 Node Package Manager의 약자입니다.
i 또는 install: Node.js 프로젝트에서 패키지를 설치하는 데 사용되는 명령입니다.
nodemon: 설치 중인 패키지의 이름입니다. 이 경우에는 nodemon 패키지를 참조합니다.
-D: --save-dev의 단축키입니다. 패키지를 package.json 파일에 개발 종속성으로 추가하도록 npm에 지시합니다.

nodemon

nodemon은 프로젝트 파일에서 변경 사항이 감지되면 Node.js 애플리케이션을 자동으로 다시 시작하여 개발 과정에서 개발자를 돕는 도구입니다. 즉, 코드를 변경할 때마다 Node.js 서버를 수동으로 중지하고 다시 시작할 필요가 없습니다. 생산성을 향상시키고 서버를 수동으로 다시 시작할 필요 없이 변경 사항을 즉시 확인할 수 있으므로 개발에 특히 유용합니다.

개발 종속성으로 nodemon을 설치하면 이 패키지가 개발 중에만 필요하고 응용 프로그램이 프로덕션에 배포될 때는 필요하지 않습니다. 프로덕션에서는 일반적으로 nodemon 없이 Node.js 애플리케이션을 실행합니다. 따라서 개발과 프로덕션 간의 종속성을 분리하여 프로덕션 환경을 간결하고 효율적으로 유지할 수 있습니다.

3.npm i @babel/core @babel/cli @babel/node -D

npm i @babel/core @babel/cli @babel/node -D 명령은 Node.js 프로젝트에서 특정 Babel 패키지를 개발 종속성으로 설치하는 데 사용됩니다.

npm: Node.js 프로젝트에서 패키지(라이브러리)를 설치하고 관리할 수 있는 도구인 Node Package Manager의 약자입니다.
i 또는 install: Node.js 프로젝트에서 패키지를 설치하는 데 사용되는 명령입니다.
@babel/core, @babel/cli, @babel/node: 설치 중인 Babel 패키지의 이름입니다.
-D: --save-dev의 단축키입니다. npm에게 package.json 파일에 이러한 패키지를 개발 종속성으로 추가하도록 지시합니다.

babel

Babel은 개발자가 최신 JavaScript 코드(최신 ECMAScript 기능 사용)를 작성한 다음 이전 브라우저나 환경에서 실행할 수 있는 이전 버전과 호환되는 JavaScript 버전으로 변환할 수 있는 인기 있는 JavaScript 컴파일러입니다. 개발자가 호환성 문제를 걱정하지 않고 최첨단 JavaScript 기능을 사용할 수 있도록 도와줍니다.

설치되는 패키지는 무엇입니까?

@babel/core: 이 패키지는 Babel의 핵심 구성 요소입니다. JavaScript 코드를 구문 분석, 변환 및 생성하기 위한 기본 기능을 제공합니다.
@babel/cli: 이 패키지에는 Babel용 명령줄 인터페이스(CLI)가 포함되어 있습니다. 이를 통해 개발자는 명령줄에서 Babel을 사용하여 JavaScript 파일을 변환할 수 있습니다.
@babel/node: 이 패키지를 사용하면 Node.js 런타임과 함께 Babel을 사용할 수 있습니다. 이는 Babel을 사용하여 Node.js 스크립트를 실행하는 방법을 제공합니다. 즉, 먼저 수동으로 변환하지 않고 최신 구문으로 JavaScript 파일을 직접 실행할 수 있습니다.

개발 의존성으로 설치되는 이유는 무엇입니까?

Babel은 주로 개발 단계에서 최신 JavaScript 코드를 이전 버전과 호환되는 버전으로 변환하는 데 사용됩니다. 프로젝트가 프로덕션 준비가 되면 변환된 코드는 일반적으로 Webpack 또는 Rollup과 같은 다른 프로덕션 도구와 함께 번들로 제공됩니다.

Babel은 최신 JavaScript 기능을 사용하기 위해 개발 중에만 필요하므로 개발 종속 항목으로 설치됩니다. 프로덕션에서는 트랜스파일된 코드를 사용하며 프로덕션 환경에 Babel을 포함할 필요가 없습니다. 프로덕션 종속성과 개발 종속성을 분리하면 프로덕션 환경을 경량화하고 최적화하는 데 도움이 됩니다.

4.nodemon.json 구성
{
    "exec": "babel-node src/server.js"
}

nodemon.json 파일에서 "exec": "babel-node src/server.js" 구성은 nodemon이 Node.js 애플리케이션을 실행하는 데 사용할 명령을 지정합니다.
"exec":: nodemon에 응용 프로그램을 실행하는 데 사용할 명령을 알려주는 nodemon.json 파일의 구성 속성입니다.
"babel-node src/server.js": 애플리케이션을 실행하기 위해 nodemon에 지정된 명령입니다. 두 부분으로 구성됩니다.
babel-node: Babel Node.js CLI를 가리키는 부분입니다. 최신 ECMAScript 기능(ES6+ 구문)이 포함된 JavaScript 파일을 Babel을 사용하여 이전 버전과 호환되는 버전으로 즉석에서 변환하여 실행할 수 있습니다.
src/server.js: 이 부분은 Node.js 애플리케이션의 진입점을 지정합니다. 이 예에서 진입점은 이름이 server.js인 src 폴더에 있는 파일입니다. 이것은 nodemon이 Babel을 사용하여 실행할 파일입니다.

개발을 위해 Babel을 babel-node와 함께 사용하는 이유는 무엇입니까?

개발 중에 babel-node를 사용하면 개발 환경의 호환성 문제에 대한 걱정 없이 최신 ECMAScript 기능을 사용하여 서버 측 JavaScript 코드를 작성할 수 있습니다. Babel은 최신 구문을 Node.js 환경에서 실행할 수 있는 버전으로 자동 변환합니다.

개발 중에 일반 node 명령 대신 babel-node를 사용하면 사용 중인 Node.js 버전에서 지원하지 않을 수 있는 최신 JavaScript 기능을 활용할 수 있다는 이점이 있습니다. 애플리케이션이 프로덕션 준비가 되면 일반적으로 빌드 프로세스를 사용하여 Babel을 사용하여 코드를 트랜스파일한 다음 Babel의 런타임 변환으로 인한 약간의 오버헤드를 피하기 위해 babel-node를 사용하는 대신 node로 트랜스파일된 코드를 실행합니다.

5.npm i @babel/preset-env

6.babel.config.json 구성
{
    "presets": ["@babel/preset-env"]
}

"presets":: 이것은 babel.config.json 파일에 있는 구성 옵션으로, Babel이 코드를 변환하는 데 사용할 사전 설정(플러그인 그룹) 목록을 지정합니다.

["@babel/preset-env"]: 구성에서 사용 중인 프리셋입니다. "@babel/preset-env" 프리셋은 Babel이 대상 환경에 따라 트랜스파일해야 하는 ECMAScript 기능(ES6+, ES7 등)을 자동으로 결정할 수 있는 강력한 프리셋입니다.

"@babel/preset-env"의 기능은 무엇인가요?

babel/preset-env는 Babel에서 제공하는 스마트하고 유연한 프리셋입니다. 이를 통해 호환성 문제에 대한 걱정 없이 최신 ECMAScript 기능을 사용하여 최신 JavaScript 코드를 작성할 수 있습니다. 이 프리셋을 사용하여 코드를 트랜스파일링할 때 Babel은 대상 환경(예: 브라우저 또는 Node.js 버전)을 분석하고 해당 환경에서 코드가 작동하도록 하는 데 필요한 변환만 적용합니다.

예를 들어, 대부분의 최신 브라우저에서 지원되는 ES6+ 기능을 사용하는 경우 Babel은 해당 기능을 지원하지 않는 구형 브라우저에서는 해당 기능을 동등한 ES5 코드로 트랜스파일하지만 이미 지원되는 기능은 트랜스파일하지 않습니다. 이렇게 하면 코드 변환을 최소화하고 최적화하여 보다 효율적인 결과물을 얻을 수 있습니다.

"@babel/preset-env"를 사전 설정으로 사용하는 것은 특히 최신 자바스크립트 개발에서 일반적인 관행으로, 이미 지원되는 기능에 대한 폴리필로 출력을 불필요하게 부풀리지 않고도 바벨 구성을 단순화하고 코드가 대상 환경과 호환되도록 보장하기 때문입니다.
